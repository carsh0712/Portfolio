GoDaddy Hosting

MySQL 연결 종료 오류 대응

GoDaddy 웹호스팅 서버에서 /api/v1/auth/login 요청 중 MySQL server has gone away와 Connection reset by peer 오류가 발생했습니다. 로그인 로직의 비밀번호 검증 문제가 아니라, SQLAlchemy 커넥션 풀에 남아 있던 오래된 MySQL 연결을 재사용하면서 발생한 DB 연결 오류입니다.

상황

  • Flask 앱은 요청 시작 시 SessionLocal()로 DB 세션을 생성합니다.
  • SQLAlchemy는 기본적으로 DB 연결을 풀에 보관했다가 다음 요청에서 재사용합니다.
  • GoDaddy 같은 공유 웹호스팅의 MySQL은 유휴 연결을 먼저 끊을 수 있습니다.
  • 이미 끊긴 연결로 로그인 사용자 조회 쿼리를 보내면서 OperationalError 2006이 발생했습니다.

적용 조치

  • pool_pre_ping=True로 쿼리 전 연결 생존 여부를 확인하도록 설정했습니다.
  • pool_recycle=1800으로 30분 이상 지난 연결은 새 연결로 교체합니다.
  • DB_POOL_RECYCLE 환경 변수로 운영 서버의 MySQL timeout에 맞춰 값을 조정할 수 있습니다.
  • DB_SQL_ECHO 환경 변수로 운영 SQL 로그 출력을 필요할 때만 켤 수 있게 했습니다.

변경 코드

engine = create_engine(
    DATABASE_URL,
    pool_pre_ping=True,
    pool_recycle=POOL_RECYCLE,
    echo=SQL_ECHO,
)
결과: 오래된 MySQL 연결을 재사용하기 전에 SQLAlchemy가 연결 상태를 점검하고, 끊어진 연결이면 새 연결로 교체합니다.
Client Performance

Lazy Import 정리

lazy import는 처음 앱을 열 때 모든 페이지 코드를 한 번에 내려받지 않고, 실제로 해당 페이지가 필요해졌을 때 JavaScript chunk를 나누어 불러오는 방식입니다.

왜 필요한가

  • 초기 로딩 시 내려받는 JavaScript 크기를 줄일 수 있습니다.
  • 사용자가 방문하지 않은 페이지 코드는 나중에 필요할 때만 로드됩니다.
  • Vite의 500kB 이상 chunk 경고를 줄이는 데 도움이 됩니다.

이번 적용

  • src/App.tsx의 페이지 import를 React.lazy()로 변경했습니다.
  • Suspense fallback으로 페이지 chunk 로딩 중 상태를 보여줍니다.
  • vite.config.ts에서 React 계열 라이브러리를 react-vendor chunk로 분리했습니다.

예시

import { lazy, Suspense } from 'react';

const ProjectDetail = lazy(() => import('./pages/ProjectDetail'));

function App() {
  return (
    <Suspense fallback={<PageState loading message="Loading page..." />}>
      <ProjectDetail />
    </Suspense>
  );
}
