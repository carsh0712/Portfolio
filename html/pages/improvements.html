개선 포인트

정적 코드 분석 과정에서 확인한 공통, 클라이언트, 서버 개선 사항을 분리했습니다.

요약 공통 정보 클라이언트 서버 개선 포인트
Common

공통 개선

인코딩 깨짐: 여러 Python/TypeScript 파일의 한국어 주석과 사용자 메시지가 깨진 문자열로 보입니다. 기능 로직은 읽히지만, 사용자에게 보이는 에러 메시지와 문서 주석은 UTF-8 기준으로 복구하는 것이 좋습니다.
  • 클라이언트의 Category 용어와 서버의 Portfolio 용어를 하나로 맞추면 유지보수성이 좋아집니다.
  • 클라이언트 ProjectDetail 라우트 파라미터 :id는 실제로 project code로 쓰이므로 :projectCode로 바꾸는 편이 명확합니다.
  • 문서 작성 과정에서 테스트는 실행하지 않았습니다. 현재 문서는 정적 코드 분석 기준입니다.
Client

클라이언트 개선

  • README가 기본 Vite 템플릿 설명에 가깝습니다. 실제 앱 실행법, 환경변수, API 서버 연결법을 보강하는 것이 좋습니다.
  • Project.links는 camelCase와 snake_case를 모두 허용하고 있어 응답 표준을 통일하면 변환 코드가 줄어듭니다.
  • 공개 페이지와 관리 페이지의 이미지 URL 생성 규칙이 분리되어 있으므로, 이미지 접근 권한 기준을 문서화하면 혼동을 줄일 수 있습니다.
Server

서버 개선

  • portfolio_project_server_fastapi라는 이름과 README 제목은 실제 Flask 구현과 맞지 않습니다.
  • database.py에 기본 DB 비밀번호가 코드에 들어 있습니다. 개발 편의값이라도 환경변수 중심으로 정리하는 편이 안전합니다.
  • JWT_SECRET_KEY 기본값이 fallback-secret-change-me입니다. 운영 환경에서는 반드시 강한 시크릿을 환경변수로 설정해야 합니다.
  • 공개 파일 API는 username과 file_uuid 소유권만 확인합니다. 특정 파일이 실제 공개 portfolio/project에서 참조되는지까지 확인하지는 않습니다.
  • 검색은 JSON 필드를 문자열로 cast한 뒤 LIKE를 사용합니다. 규모가 커지면 MySQL JSON 검색 또는 별도 색인 전략이 필요합니다.
현재 코드의 큰 방향: 개인 관리용 CRUD 화면과 공개 포트폴리오 화면이 모두 준비되어 있고, 서버도 인증/소유권/공개 여부를 기준으로 API 경계를 나누고 있습니다.
