# Day 7. 포트폴리오에 작성자 프로필 연결하기

## 시작하며

포트폴리오에는 프로젝트 목록만 있는 것보다 작성자 정보가 함께 있는 편이 훨씬 자연스럽다.

오늘은 다중 프로필 기능을 추가했다. 사용자는 여러 프로필을 만들 수 있고, 포트폴리오마다 특정 프로필을 연결할 수 있다.

## 오늘의 목표

- `profile` 테이블을 추가한다.
- 사용자가 여러 프로필을 만들 수 있게 한다.
- 기본 프로필 개념을 넣는다.
- 포트폴리오에 프로필을 연결한다.
- 공개 페이지에서 프로필 다이얼로그를 보여준다.

## 프로필 모델

프로필에는 다음 정보를 담았다.

- 표시 이름
- 이메일
- 한 줄 소개
- 소개글
- 아바타 이미지
- 링크 목록
- 유동 항목
- 기본 프로필 여부

유동 항목은 프로필마다 다른 정보를 넣을 수 있도록 JSON 배열로 구성했다. 예를 들어 역할, 위치, 가능한 작업, 관심 기술 같은 값을 사용자가 자유롭게 추가할 수 있다.

## 포트폴리오와 프로필 연결

포트폴리오에는 `profile_id`를 추가했다.

```text
portfolio.profile_id -> profile.id
```

프로필이 삭제되면 포트폴리오의 연결은 `SET NULL`로 풀리도록 했다. 포트폴리오 자체가 함께 삭제되면 안 되기 때문이다.

## 공개 프로필

공개 포트폴리오 화면에는 "프로필 보기" 버튼을 추가했다. 버튼을 누르면 다이얼로그가 열리고, 프로필 이미지, 소개, 이메일, 링크, 유동 항목을 확인할 수 있다.

프로필이 포트폴리오에 연결되어 있으면 연결된 프로필을 사용하고, 연결된 프로필이 없으면 기본 프로필을 사용할 수 있게 했다.

## 고민한 점

프로필 정보를 포트폴리오에 직접 넣을 수도 있었다. 하지만 그렇게 하면 포트폴리오가 여러 개일 때 같은 작성자 정보를 반복해서 수정해야 한다.

그래서 프로필을 별도 리소스로 만들었다. 이 구조에서는 포트폴리오마다 다른 프로필을 연결할 수도 있고, 하나의 기본 프로필을 여러 포트폴리오에서 재사용할 수도 있다.

## 문제와 해결

프로필 링크와 유동 항목은 정해진 필드가 아니라 배열 데이터다. 이 값을 어떻게 검증하고 저장할지가 고민이었다.

초기에는 유연성을 우선해 JSON으로 저장했다. 대신 공개 API에서는 `is_public` 값이 있는 유동 항목만 내려주도록 필터링했다.

## 오늘의 결과

포트폴리오에 작성자 정보가 붙으면서 공개 페이지가 더 완성된 형태에 가까워졌다.

이제 방문자는 프로젝트뿐 아니라 작성자의 소개와 연락 링크도 함께 확인할 수 있다.

## 다음 작업

다음은 UI 정리다. 프로필 관리 화면에서 너무 많은 정보가 카드에 노출되면 관리 화면이 복잡해진다. 정보 밀도를 조정하고, 공개 화면의 링크 스타일도 더 자연스럽게 다듬을 예정이다.
