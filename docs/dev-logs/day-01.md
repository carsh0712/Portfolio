# Day 1. 포트폴리오 관리 서비스의 뼈대 만들기

## 시작하며

이번 프로젝트의 목표는 단순한 정적 포트폴리오 페이지를 넘어서, 사용자가 직접 포트폴리오와 프로젝트를 관리하고 공개 페이지로 보여줄 수 있는 서비스를 만드는 것이다.

처음부터 가장 중요하게 본 것은 "관리 화면"과 "공개 화면"을 분리하는 일이었다. 관리자는 로그인 후 포트폴리오와 프로젝트를 편집하고, 방문자는 별도의 공개 URL을 통해 정리된 결과만 볼 수 있어야 했다.

## 오늘의 목표

- React/Vite 기반 클라이언트와 Flask 기반 서버 구조를 나눈다.
- MySQL을 기준으로 데이터 저장 구조를 설계한다.
- 사용자, 포트폴리오, 프로젝트를 핵심 도메인으로 잡는다.
- 이후 기능을 붙이기 쉬운 폴더 구조를 만든다.

## 프로젝트 구조 잡기

프로젝트는 크게 두 영역으로 나눴다.

```text
portfolio_project_client_vite/
portfolio_project_server_flask/
```

클라이언트는 React와 Vite를 사용했다. 관리 화면과 공개 화면 모두 SPA 라우팅이 필요했고, 빠르게 화면을 추가하고 테스트하기에는 Vite 구성이 가볍고 적당했다.

서버는 Flask를 사용했다. API 라우터를 기능별로 나누고, SQLAlchemy 모델을 중심으로 MySQL과 연결하는 방식으로 구성했다. 프로젝트 규모가 아주 크지는 않지만, 인증, 업로드, 공개 API, 관리 API가 분리될 예정이었기 때문에 처음부터 라우터를 나누는 편이 낫다고 판단했다.

## 핵심 도메인 설계

처음 잡은 핵심 도메인은 세 가지였다.

- User: 서비스를 사용하는 계정
- Portfolio: 사용자가 공개할 수 있는 포트폴리오 묶음
- Project: 포트폴리오 안에 들어가는 개별 프로젝트

Portfolio에는 `code`를 두어 공개 URL에 사용할 수 있게 했다. Project에도 `code`를 두어 상세 페이지가 숫자 ID에 의존하지 않도록 했다.

```text
/public/:username/:portfolioCode
/public/:username/:portfolioCode/:projectCode
```

이렇게 URL을 설계하면 나중에 DB의 내부 ID가 바뀌어도 공개 주소는 사람이 읽기 쉬운 형태로 유지할 수 있다.

## 환경 변수와 DB 연결

서버는 환경 변수로 DB 접속 정보를 받도록 구성했다.

```text
MYSQL_HOST
MYSQL_PORT
MYSQL_DB
MYSQL_USER
MYSQL_PASSWORD
```

개발 환경에서는 Docker Compose로 MySQL을 띄울 수 있게 하고, 운영 환경에서는 호스팅 서버의 DB 정보를 넣어 연결할 수 있게 열어두었다.

DB 연결은 SQLAlchemy `create_engine`을 사용했고, 운영 환경에서 오래 열린 커넥션이 끊기는 상황을 대비해 `pool_pre_ping`, `pool_recycle` 설정도 고려했다.

## 오늘 고민한 점

가장 큰 고민은 "처음부터 얼마나 일반화할 것인가"였다.

포트폴리오 서비스라고 하면 카테고리, 섹션, 템플릿, 테마 같은 기능을 처음부터 많이 상상하게 된다. 하지만 초반부터 구조를 크게 벌리면 실제로 동작하는 화면을 늦게 보게 된다.

그래서 첫날에는 기능을 넓히기보다 다음 기준만 지키기로 했다.

- 사용자별 데이터가 분리될 것
- 공개 URL이 안정적일 것
- 프로젝트 상세 정보를 나중에 확장할 수 있을 것
- 클라이언트와 서버가 느슨하게 연결될 것

## 오늘의 결과

오늘은 실제 기능을 많이 만든 날이라기보다, 이후 기능을 얹을 수 있는 바닥을 만든 날이었다.

프로젝트는 React/Vite 클라이언트와 Flask 서버로 나뉘었고, MySQL을 중심으로 사용자, 포트폴리오, 프로젝트 모델을 확장할 준비를 마쳤다.

## 다음 작업

다음 단계는 인증이다. 포트폴리오 관리 기능은 사용자별로 보호되어야 하므로 회원가입, 로그인, 토큰 발급, 보호 라우트를 먼저 구현할 계획이다.
