# Day 10. 테스트로 주요 기능 안정화하기

## 시작하며

기능을 빠르게 붙이다 보면 어느 순간부터 "잘 돌아가는 것 같다"와 "정말 깨지지 않는다" 사이의 간격이 커진다.

오늘은 그 간격을 줄이기 위해 테스트와 빌드 검증을 정리했다.

## 오늘의 목표

- 프론트엔드 테스트를 실행한다.
- 백엔드 API 테스트 범위를 확인한다.
- 공개 API와 관리 API의 주요 흐름을 점검한다.
- TypeScript 빌드와 Vite production build를 확인한다.

## 프론트엔드 테스트

클라이언트는 Vitest 기반 테스트를 사용한다.

테스트 대상에는 API 유틸, 아이콘, 모달, 토스트, 프로젝트 카드, 프로젝트 링크, 컬러 피커 같은 단위가 포함된다.

특히 링크 스타일 테스트는 중요했다. 프로젝트 링크와 프로필 링크는 배경색/글자색/아이콘을 사용자 설정으로 받기 때문에, 스타일 적용이 깨지면 공개 화면의 느낌이 바로 달라진다.

## 빌드 검증

프론트엔드는 테스트뿐 아니라 production build도 확인했다.

```text
npm run build
```

이 과정에서는 TypeScript 빌드와 Vite 번들링이 함께 실행된다. 단순 렌더링 테스트에서 놓칠 수 있는 import 오류나 타입 오류를 잡는 데 도움이 된다.

## 백엔드 테스트

백엔드에는 인증, 포트폴리오, 프로젝트, 업로드, 프로필, 공개 API 관련 테스트가 있다.

중요하게 본 흐름은 다음과 같다.

- 로그인과 토큰 발급
- 사용자별 리소스 분리
- 포트폴리오 CRUD
- 프로젝트 CRUD
- 공개 프로젝트 조회
- 이미지 업로드와 파일 접근
- 프로필 생성과 연결
- 공개 프로필 조회

API 테스트는 특히 권한 경계를 확인하는 데 유용하다. 내 데이터만 수정할 수 있는지, 비공개 데이터가 공개 API에 섞이지 않는지 계속 확인해야 한다.

## UI 변경 후 검증

최근에는 프로필 관리 카드에서 유동 항목과 소개글을 숨기고, 공개 프로필 링크에 색상과 아이콘을 적용했다.

이런 변경은 작아 보여도 JSX 구조나 import가 깨질 수 있다. 그래서 수정 후에는 Vitest와 production build를 함께 돌려 확인했다.

## 고민한 점

테스트는 많을수록 좋지만, 모든 화면 상태를 처음부터 완벽하게 테스트하려고 하면 속도가 떨어진다.

이번 단계에서는 다음 기준으로 우선순위를 잡았다.

- 데이터 변환 로직
- 공개/비공개 경계
- 인증이 필요한 API
- 반복 사용되는 UI 컴포넌트
- 빌드 단계에서 잡히는 타입 오류

화면 전체의 시각적 회귀 테스트는 나중에 Playwright 기반으로 확장할 수 있다.

## 오늘의 결과

주요 기능이 테스트와 빌드를 통과하는지 확인했다.

서비스는 이제 인증, 포트폴리오 관리, 프로젝트 관리, 이미지 업로드, 공개 페이지, 프로필 기능까지 기본 흐름을 갖췄고, 변경 후 검증하는 루틴도 어느 정도 정리되었다.
