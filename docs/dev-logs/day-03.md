# Day 3. 포트폴리오 관리 기능 만들기

## 시작하며

인증 흐름이 만들어졌으니 이제 사용자가 실제로 관리할 수 있는 첫 번째 리소스를 붙일 차례다.

오늘은 포트폴리오 CRUD를 구현했다. 포트폴리오는 프로젝트들을 담는 상위 단위이고, 공개 페이지의 기준이 되는 리소스다.

## 오늘의 목표

- 포트폴리오 생성, 목록, 수정, 삭제 API를 만든다.
- 사용자별로 포트폴리오를 분리한다.
- 공개 URL에 사용할 `code` 필드를 둔다.
- 클라이언트에서 포트폴리오 목록과 작성 화면을 연결한다.

## 포트폴리오 모델

포트폴리오에는 다음 정보를 담았다.

- `user_id`: 소유자
- `code`: 공개 URL에 사용할 코드
- `name`: 포트폴리오 이름
- `description`: 설명
- `order`: 정렬 순서
- `is_public`: 공개 여부

특히 `code`는 사용자별로 유니크하게 유지되도록 했다.

```text
user_id + code unique
```

이렇게 하면 서로 다른 사용자가 같은 `code`를 써도 충돌하지 않고, 한 사용자 안에서는 공개 URL이 중복되지 않는다.

## API 구현

관리 API는 로그인한 사용자 기준으로 동작한다.

```text
GET    /api/v1/portfolios/
POST   /api/v1/portfolios/
GET    /api/v1/portfolios/:id
PUT    /api/v1/portfolios/:id
DELETE /api/v1/portfolios/:id
```

조회, 수정, 삭제 시에는 반드시 현재 로그인한 사용자의 포트폴리오인지 확인한다. 이 확인을 빼먹으면 다른 사용자의 데이터를 건드릴 수 있기 때문에, CRUD 구현에서 가장 중요하게 봤다.

## 클라이언트 화면

클라이언트에는 포트폴리오 목록 화면과 추가/수정 화면을 구성했다.

목록 화면에서는 포트폴리오 이름, 설명, 공개 상태를 카드 형태로 보여준다. 사용자는 카드를 통해 상세 관리 화면으로 들어가거나 새 포트폴리오를 만들 수 있다.

## 고민한 점

삭제 정책을 어떻게 둘지가 고민이었다.

포트폴리오를 삭제하면 그 안의 프로젝트도 함께 삭제되는 것이 자연스럽다. SQLAlchemy 관계에서는 cascade를 사용해 이 흐름을 모델에 반영했다.

다만 실제 서비스라면 삭제 전에 연결된 프로젝트 수를 보여주거나, 휴지통처럼 되돌릴 수 있는 정책도 고려할 수 있다. 지금 단계에서는 단순하고 명확한 삭제 정책을 선택했다.

## 오늘의 결과

사용자는 이제 자신의 포트폴리오를 만들고 관리할 수 있다.

서비스가 단순한 로그인 화면에서 실제 관리 도구로 넘어가기 시작한 날이었다. 포트폴리오라는 상위 컨테이너가 생겼기 때문에, 다음에는 그 안에 들어갈 프로젝트 정보를 설계할 수 있다.
