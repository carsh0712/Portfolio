# Day 6. 외부에 보여줄 공개 포트폴리오 화면 만들기

## 시작하며

관리 화면에서 데이터를 잘 입력할 수 있게 되었으니, 이제 방문자가 보는 화면을 만들 차례다.

오늘은 공개 포트폴리오 페이지와 공개 프로젝트 상세 페이지를 구현했다. 이 기능은 서비스의 결과물이 실제로 보이는 부분이라, 관리 기능만큼 중요하다.

## 오늘의 목표

- 로그인 없이 접근 가능한 공개 포트폴리오 URL을 만든다.
- 공개 상태인 포트폴리오와 프로젝트만 노출한다.
- 공개 프로젝트 목록과 상세 화면을 구성한다.
- 태그 기반 필터링을 제공한다.
- 공개 API와 관리 API의 책임을 분리한다.

## 공개 URL 설계

공개 URL은 사용자명과 포트폴리오 코드를 기준으로 구성했다.

```text
/public/:username/:portfolioCode
/public/:username/:portfolioCode/:projectCode
```

숫자 ID가 아닌 사람이 읽을 수 있는 값으로 URL을 만들면 공유했을 때도 의미가 드러난다.

예를 들어 아래와 같은 주소가 가능하다.

```text
/public/admin/MYAPPS
```

## 공개 API 구현

공개 API는 로그인 없이 접근 가능하지만, 모든 데이터를 보여주면 안 된다.

그래서 다음 기준을 적용했다.

- 공개 포트폴리오만 조회 가능
- 공개 프로젝트만 목록에 포함
- 비공개 프로젝트 상세 접근 차단
- 공개 페이지에서 참조하는 이미지 파일만 노출

관리 API에서는 현재 사용자 기준으로 데이터를 다루고, 공개 API에서는 `username`, `portfolioCode`, `projectCode` 기준으로 데이터를 찾는다.

## 클라이언트 화면

공개 포트폴리오 화면에서는 포트폴리오 이름과 설명을 상단에 보여주고, 프로젝트 목록을 카드 형태로 배치했다.

프로젝트가 많아질 수 있으므로 태그 필터링도 추가했다. 사용자는 특정 태그를 선택해 관련 프로젝트만 볼 수 있다.

프로젝트 상세 화면에서는 설명, 기술 스택, 주요 기능, 스크린샷, 외부 링크를 보여준다.

## 고민한 점

가장 중요한 고민은 공개 화면에서 어느 정보를 숨길 것인가였다.

관리 화면의 데이터가 모두 공개되어도 되는 것은 아니다. 그래서 `is_public` 값을 기준으로 공개 여부를 명확히 나누었다.

또 하나는 공개 파일 접근이었다. 파일 URL이 공개되어도, 그 파일이 공개 프로젝트에서 참조되는지 확인해야 한다. 이 검증이 있어야 비공개 프로젝트의 이미지가 우연히 노출되는 일을 줄일 수 있다.

## 오늘의 결과

이제 서비스는 입력 도구를 넘어, 실제로 외부에 공유할 수 있는 포트폴리오 페이지를 갖게 되었다.

관리 화면에서 만든 포트폴리오와 프로젝트가 공개 URL을 통해 방문자에게 전달된다.

## 다음 작업

다음은 프로필 기능이다. 포트폴리오에는 프로젝트뿐 아니라 작성자 정보도 필요하다. 프로필을 만들고, 포트폴리오에 연결하고, 공개 페이지에서 보여줄 수 있게 할 예정이다.
