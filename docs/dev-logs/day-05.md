# Day 5. 이미지 업로드와 썸네일 처리 구현

## 시작하며

포트폴리오에서 이미지는 생각보다 중요하다. 프로젝트 설명이 아무리 좋아도, 목록 카드와 상세 화면에서 시각적인 단서가 없으면 결과물이 잘 전달되지 않는다.

오늘은 이미지 업로드 API와 썸네일 처리 흐름을 구현했다.

## 오늘의 목표

- 이미지 업로드 API를 만든다.
- 업로드 파일의 메타데이터를 DB에 저장한다.
- 원본 대신 WebP 변환 이미지를 사용한다.
- 목록에는 썸네일, 상세에는 더 큰 이미지를 제공한다.
- 공개 페이지에서도 안전하게 이미지를 조회할 수 있게 한다.

## 업로드 파일 모델

업로드된 파일은 `upload_file` 테이블에 메타데이터를 저장한다.

- `uuid`
- `user_id`
- `original_filename`
- `stored_filename`
- `file_size`
- `content_type`
- `upload_path`
- `created_at`

파일을 직접 ID로 노출하지 않고 UUID를 사용했다. 공개 URL에서 파일을 참조할 때 내부 auto increment ID보다 UUID가 더 안전하고 다루기 좋기 때문이다.

## 이미지 처리 정책

이미지 업로드는 JPG, PNG, WebP를 허용하고 GIF는 제외했다.

업로드 후에는 서버에서 WebP로 변환하고, 용도에 따라 두 가지 크기를 만든다.

- thumbnail: 목록 카드와 작은 미리보기용
- detail: 상세 화면과 라이트박스용

이렇게 나누면 공개 목록에서 큰 이미지를 매번 불러오지 않아도 된다.

## 클라이언트 적용

클라이언트에서는 업로드 후 반환된 파일 UUID를 프로젝트나 프로필 데이터에 저장한다.

프로젝트 목록 카드에서는 thumbnail 변형을 사용하고, 상세 화면이나 라이트박스에서는 detail 변형을 사용하도록 나누었다.

이 방식 덕분에 같은 파일을 여러 화면에서 다른 크기로 활용할 수 있다.

## 공개 파일 접근

파일 API는 공개 포트폴리오와 연결된 이미지인지 확인해야 한다. 아무 파일 UUID나 알면 다운로드할 수 있는 구조는 위험하기 때문이다.

그래서 공개 파일 조회는 공개 포트폴리오/프로젝트에서 참조되는 파일인지 확인한 뒤 제공하는 방향으로 설계했다.

## 고민한 점

처음에는 업로드한 원본 파일을 그대로 보여주는 방법도 가능했다. 하지만 원본 이미지는 크기가 제각각이고, 모바일 환경에서 불필요하게 무거울 수 있다.

그래서 서버에서 변환과 리사이징을 담당하게 했다. 클라이언트는 어떤 변형이 필요한지만 요청하면 된다.

## 오늘의 결과

프로젝트에 실제 이미지를 붙일 수 있게 되었고, 목록과 상세 화면에서 필요한 크기의 이미지를 나누어 사용할 수 있게 되었다.

포트폴리오가 텍스트 중심에서 시각적인 결과물 중심으로 한 단계 올라간 느낌이다.
