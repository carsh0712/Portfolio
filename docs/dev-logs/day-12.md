# Day 12. MCP 서버 개발

## 시작하며

포트폴리오 관리 API는 어느 정도 기능이 갖춰졌지만, 개발 중에 데이터를 만들고 확인하는 흐름은 아직 꽤 수동적이었다.

포트폴리오를 하나 만들고, 프로젝트를 추가하고, 파일을 업로드하고, 공개 응답을 다시 확인하는 작업이 계속 반복되다 보니 브라우저와 API 클라이언트를 오가는 비용이 점점 커졌다. 오늘은 이 흐름을 줄이기 위해 포트폴리오 API를 MCP 서버로 감싸는 작업을 진행했다.

## 오늘의 목표

- 포트폴리오 API를 MCP 도구 형태로 노출한다.
- 인증, 포트폴리오, 프로젝트, 파일, 공개 조회 기능을 MCP로 호출할 수 있게 만든다.
- 서버 코드가 빠르게 비대해지지 않도록 구조를 역할별로 나눈다.
- 이후 기능 추가가 쉬운 형태로 정리한다.

## 왜 MCP 서버가 필요했는가

개발 과정에서 자주 하는 작업은 대부분 정해져 있다.

- 로그인 상태 확인
- 포트폴리오 목록 조회
- 프로젝트 생성 또는 수정
- 파일 업로드 후 UUID 확인
- 공개 프로젝트 응답 검증

이런 작업을 매번 수동으로 반복하기보다는, MCP 도구처럼 바로 호출할 수 있게 만들면 개발 피드백 루프가 훨씬 짧아진다. 오늘 작업은 바로 이 반복 비용을 줄이는 데 초점을 맞췄다.

## 구현한 내용

`portfolio_mcp_server` 패키지를 기준으로 Node 기반 MCP 서버를 구성했다.

기본 흐름은 다음과 같다.

- 서버 시작
- 도구 목록 등록
- 도구 호출 수신
- 내부 API 클라이언트로 라우팅
- JSON 또는 파일 응답 반환

도구 범위는 크게 다섯 묶음으로 나눴다.

- 인증 도구
- 포트폴리오 도구
- 프로젝트 도구
- 파일 도구
- 공개 조회 도구

API 클라이언트는 단순 fetch 래퍼를 넘어서 다음 책임을 갖도록 구성했다.

- 환경 변수 로드
- 로그인 및 토큰 메모리 보관
- 인증 실패 시 refresh 또는 재로그인 처리
- JSON 요청/응답 처리
- 파일 업로드 처리
- 파일 응답의 base64 변환 지원
- 네트워크 오류와 API 오류를 구분한 예외 처리

## 구조를 정리한 이유

MCP 서버는 처음엔 빠르게 한 파일로 만들 수 있지만, 기능이 늘면 스키마 정의, 툴 정의, 라우팅, 실제 호출 처리, 응답 포맷팅이 한 곳에 엉키기 쉽다.

그래서 오늘은 기능 구현과 함께 구조 정리도 같이 진행했다.

현재 `portfolio_mcp_server/src/mcp`는 다음처럼 나뉜다.

- `schemas`: 입력 스키마와 공통 스키마 조각
- `tools`: MCP 도구 정의
- `handlers`: 실제 호출 라우팅과 응답 처리

각 디렉터리 내부도 `auth`, `portfolio`, `project`, `file`, `public` 단위로 다시 분리해 두었다. 덕분에 이후에 새 도구를 추가할 때도 어떤 파일을 수정해야 하는지가 바로 보이게 됐다.

API 쪽도 역할을 나눴다.

- `client.ts`: 인증 흐름과 상위 API 메서드
- `http.ts`: 저수준 HTTP 처리
- `files.ts`: 파일 업로드와 파일 응답 가공
- `types.ts`, `error.ts`: 타입과 에러 정의

## 확인한 것

구조를 나눈 뒤에는 TypeScript 타입 체크로 계속 검증했다.

```bash
.\node_modules\.bin\tsc.cmd -p tsconfig.json --noEmit
```

이 과정에서 폴더 분리 후 깨지기 쉬운 상대 경로 문제도 함께 정리했다.

## 오늘의 결과

오늘 작업으로 포트폴리오 관리 API를 MCP 서버 형태로 사용할 수 있는 기반이 생겼다.

인증, 포트폴리오, 프로젝트, 파일, 공개 조회까지 한 흐름 안에서 다룰 수 있게 됐고, 구현 구조도 충분히 쪼개 두어서 다음 작업의 부담도 줄였다. 이제는 MCP 서버를 새로 만드는 단계보다는, 기존 도구를 확장하고 다듬는 단계로 넘어갈 수 있게 됐다.
