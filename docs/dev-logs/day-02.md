# Day 2. 사용자 인증과 보호 라우트 구현

## 시작하며

포트폴리오 관리 서비스에서 인증은 거의 모든 관리 기능의 출발점이다. 프로젝트와 포트폴리오는 사용자별로 분리되어야 하고, 공개 페이지를 제외한 관리 API는 로그인한 사용자만 접근할 수 있어야 한다.

오늘은 회원가입, 로그인, 토큰 발급, 보호 라우트의 기본 흐름을 구현했다.

## 오늘의 목표

- 회원가입 API를 만든다.
- 로그인 시 access token과 refresh token을 발급한다.
- 관리 API에 인증 데코레이터를 적용할 수 있게 한다.
- 클라이언트에서 로그인 상태를 유지하고 보호 라우트를 구성한다.

## 인증 흐름 설계

인증은 JWT 기반으로 구성했다.

로그인에 성공하면 서버는 두 종류의 토큰을 반환한다.

- access token: API 요청에 사용하는 짧은 수명의 토큰
- refresh token: access token을 재발급하기 위한 긴 수명의 토큰

클라이언트는 로그인 성공 후 토큰을 저장하고, 이후 관리 API 요청에 access token을 붙여 보낸다.

## 서버 구현

서버에서는 인증 관련 라우터를 별도로 분리했다.

```text
routers/auth.py
core/security.py
schemas/auth.py
```

비밀번호는 평문으로 저장하지 않고 해시해서 저장한다. 로그인 시에는 입력받은 비밀번호를 검증하고, 성공한 경우 토큰을 발급한다.

관리 API에서는 `login_required` 형태의 데코레이터를 사용할 수 있게 했다. 이 데코레이터는 요청 헤더의 토큰을 확인하고, 현재 사용자를 `g.current_user`에 담아 라우터에서 사용할 수 있게 한다.

## 클라이언트 구현

클라이언트에서는 인증 상태를 전역으로 관리하기 위해 `AuthContext`를 구성했다.

로그인 화면에서 이메일과 비밀번호를 입력하면 서버의 로그인 API를 호출하고, 성공 시 홈 화면으로 이동한다.

보호 라우트는 로그인 여부를 확인한 뒤 접근을 허용하거나 로그인 화면으로 돌려보낸다.

```text
로그인 안 됨 -> /login
로그인 됨 -> 관리 화면 접근 가능
```

## 고민한 점

초기에는 access token만 사용해도 구현은 단순하다. 하지만 실제 사용성을 생각하면 토큰 만료 때마다 다시 로그인해야 하는 흐름은 불편하다.

그래서 refresh token 구조를 함께 넣었다. 구현량은 조금 늘어나지만, 인증 흐름을 서비스답게 가져갈 수 있고 이후 로그아웃, 토큰 폐기, 세션 만료 처리도 자연스럽게 확장할 수 있다.

## 문제와 해결

토큰 인증에서 중요한 것은 실패 응답을 일관되게 만드는 일이었다. 토큰이 없거나, 만료되었거나, 잘못된 경우가 모두 다르게 발생할 수 있기 때문이다.

서버에서는 인증 실패를 명확한 HTTP 상태 코드로 반환하고, 클라이언트에서는 인증 실패 시 로그인 화면으로 보내는 흐름을 정리했다.

## 오늘의 결과

이제 사용자는 회원가입과 로그인을 할 수 있고, 로그인한 사용자만 관리 기능에 접근할 수 있다.

서비스의 "내 데이터"라는 경계가 만들어졌고, 이후 포트폴리오와 프로젝트 API는 모두 이 인증 흐름 위에 얹을 수 있게 되었다.
