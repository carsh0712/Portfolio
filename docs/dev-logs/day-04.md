# Day 4. 프로젝트 상세 정보를 풍부하게 만들기

## 시작하며

포트폴리오가 그릇이라면 프로젝트는 실제 콘텐츠다. 방문자가 포트폴리오에서 가장 오래 보는 것도 결국 프로젝트 카드와 상세 설명이다.

오늘은 프로젝트 CRUD를 만들고, 프로젝트를 단순 제목/설명 수준이 아니라 포트폴리오에 올릴 만한 상세 정보 구조로 확장했다.

## 오늘의 목표

- 프로젝트 생성, 목록, 상세, 수정, 삭제 API를 만든다.
- 프로젝트에 기술 스택, 태그, 주요 기능, 기간 정보를 추가한다.
- 외부 링크와 스크린샷을 JSON 배열로 관리한다.
- 클라이언트에서 프로젝트 카드와 상세 화면을 구성한다.

## 프로젝트 모델 확장

프로젝트에는 기본 정보 외에 여러 배열형 데이터가 필요했다.

- `tags`
- `tech_stack`
- `screenshots`
- `links`
- `features`

이 값들은 MySQL JSON 컬럼으로 저장했다. 관계형 테이블로 세세하게 나눌 수도 있지만, 초기 서비스에서는 프로젝트 상세 정보가 프로젝트에 종속된 데이터에 가깝다고 판단했다.

예를 들어 링크는 이런 형태로 저장한다.

```json
[
  {
    "name": "GitHub",
    "url": "https://github.com/example/project",
    "backgroundColor": "#181717",
    "textColor": "#ffffff",
    "icon": "github"
  }
]
```

## API 구현

프로젝트 API는 포트폴리오에 속한 리소스로 다뤘다. 생성 시에는 대상 포트폴리오가 현재 사용자의 것인지 확인하고, 수정/삭제 시에도 같은 검증을 적용했다.

프로젝트에도 `code`를 두어 공개 상세 URL에 사용할 수 있게 했다.

```text
/public/:username/:portfolioCode/:projectCode
```

내부 ID가 아니라 사람이 읽을 수 있는 코드로 접근하게 하는 것이 목표였다.

## 클라이언트 구현

클라이언트에는 프로젝트 카드, 목록, 상세 화면을 만들었다.

카드에서는 제목, 요약, 태그, 썸네일을 중심으로 보여주고, 상세 화면에서는 설명, 기술 스택, 주요 기능, 링크, 스크린샷을 더 풍부하게 보여준다.

링크는 단순 텍스트 링크가 아니라 배경색, 글자색, 아이콘을 설정할 수 있도록 했다. 나중에 공개 페이지에서 GitHub, Demo, Blog 링크가 시각적으로 구분되도록 하기 위해서다.

## 고민한 점

프로젝트 상세 정보를 어디까지 정규화할지가 가장 큰 고민이었다.

기술 스택, 태그, 링크를 각각 별도 테이블로 만들면 검색과 통계에는 유리하다. 하지만 지금 단계에서는 사용자가 프로젝트 하나를 작성하고 보여주는 흐름이 더 중요했다.

그래서 먼저 JSON 컬럼으로 빠르게 구현하고, 데이터가 커지거나 검색 요구가 강해지면 별도 테이블로 분리하는 방향을 남겨두었다.

## 오늘의 결과

프로젝트가 단순 CRUD 리소스가 아니라, 포트폴리오 콘텐츠로 보여줄 만한 구조를 갖추게 되었다.

이제 사용자는 프로젝트의 핵심 정보와 외부 링크, 스크린샷을 함께 관리할 수 있다.
